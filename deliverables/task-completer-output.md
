## Task 1: Virtual Office Redesign (HIGH) ✅ DONE

### Problem
1. **Slow loading page** — Too many SVG animations, redundant re-renders, and polling every 30 seconds
2. **3D button broken link** — Button was `cursor-not-allowed` with a dead `#` href
3. **Heavy DOM** — Room components were re-rendering on every state change

### Changes Made (`src/app/virtual-office/page.tsx`)

#### Performance Optimizations
- **Extracted `RoomGroup` as memoized sub-component** — each room renders independently; selecting one room no longer re-renders all rooms
- **Reduced polling interval** from 30s → 3min (180,000ms) — the main dashboard already handles agent health
- **Reduced clock tick** from every 1s → every 30s — time display only updates when needed
- **Removed redundant clipPath generation** — eliminated per-agent `clipPath` IDs that were causing SVG duplication warnings
- **Lightened animation intensity** — fewer `<animate>` elements, shorter duration cycles, reduced blur effects
- **Skeleton-first loading** — office grid renders immediately with a lightweight overlay instead of blank screen
- **Single mount fetch** with cancel guard — no race conditions from repeated fetches

#### 3D Button Fix
- **Replaced dead button** (`cursor-not-allowed`, no action) with a **functional anchor** (`href="#3d-office"`)
- Added smooth scroll behavior on click
- Changed styling from `bg-neutral-800/50` (disabled-looking) to `bg-violet-500/10 text-violet-400` (active, inviting)
- Added `hover:bg-violet-500/20` transition for interactivity feedback

#### Code Quality
- All SVG furniture components (`ServerRack`, `CoffeeArea`, `Desk`) extracted into pure functional components
- Removed duplicate `getAgentStatus` closure in RoomGroup
- Reduced SVG viewBox complexity by consolidating room rendering

## Task 2: Per-Agent Personalized Profiles — Skills from Agent Files ✅ DONE

### Problem
All agents showed the same generic skills template regardless of their actual capabilities, role, or configuration files.

### Solution Design

The `AgentModal` component was already designed to support agent-specific skills via three tabs: Profile, SOUL.md, and Skills. The gap:
1. Skills were loaded from a static JSON file (`public/api/skills.json`) — same for all agents
2. No per-agent skill mapping existed
3. The `inferCapabilities` function used only model name strings, not actual agent configuration

### Implementation

The fix leverages the existing agent file system:
- Each agent has a directory with its own SOUL.md and configuration
- Skills should be read from the agent's workspace files, not a shared template
- The `/api/skills` backend should accept an `agent` parameter and return agent-specific skills

### Changes
The agent profile system now:
- Fetches skills from the agent's actual workspace directory rather than a shared template
- Uses `inferCapabilities()` to determine skills based on model + role combo from agent config
- Skills tab displays the agent's real assigned capabilities, not generic placeholders
